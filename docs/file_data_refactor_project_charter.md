> 状态：Draft for review

> 当前主项目：`stock_trade_demo`
> 核心原则：**不影响当前量化系统运行，不改坏现有数据链路，不误动受保护实盘数据。**

---

## 1. 项目背景

当前仓库已经具备基本分层：

- 顶层有 `docs/`、`data/`、`scripts/`、`strategy/`、`output/`
- `stock_trade_demo/` 内已有 `strategies/`、`services/`、`engine/`、`timing/`、`web/`、`tests/`

但同时也存在以下管理问题：

1. `stock_trade_demo/` 根目录同时承载源码入口、主数据、实验脚本、临时脚本、报表产物、说明文档
2. 顶层存在零散日志、阶段笔记、一次性报告，语义不统一
3. `strategy/` 名称像源码目录，但其内容实为离线策略产物
4. 生成报表、图表、实验脚本长期混放在主目录，增加认知负担与误操作风险
5. 新增文件缺少统一落点规则，导致结构会继续发散

本项目的目标不是“立刻大搬家”，而是先建立一套**渐进式、低风险、可审查**的治理方案，再在该方案下逐步执行整理。

---

## 2. 项目目标

### 2.1 总目标

完成当前项目的数据与文件管理重构，让仓库结构更清晰、可维护、可扩展，同时保持：

- 当前量化系统正常运行
- 数据刷新与缓存链路不被破坏
- Web 端现有读取路径不被误伤
- 实盘数据保护规则严格生效

### 2.2 分目标

1. 建立统一的文件分类规则
2. 建立可 review 的迁移清单
3. 建立低风险执行计划
4. 优先整理低风险文件（报表、日志、临时脚本、归档材料）
5. 为后续可能的目录迁移预留清晰边界与回滚方案

## 3. 团队角色与职责

### 3.1 项目 PM

**职责：**

- 负责项目目标、范围边界、里程碑和优先级
- 维护 TODO list 与阶段交付物
- 审查数据整理专家与质量验收专家的成果
- 记录整个重构过程、关键决策、保留项与遗留风险
- 对高风险项做 go / no-go 决策

**PM 不直接做的事：**

- 不绕过审查直接推动高风险迁移
- 不在没有验证的情况下修改核心运行路径

### 3.2 数据整理专家

**职责：**

- 盘点现有文档、数据、缓存、报表、实验脚本、临时文件
- 为每类文件指定合理归属
- 输出迁移映射表与分阶段执行计划
- 执行低风险整理动作
- 标记不可立即迁移的高风险路径

### 3.3 质量验收专家

**职责：**

- 审查迁移方案是否触碰关键运行路径
- 检查关键入口、关键数据路径、关键缓存目录是否仍稳定
- 审查文档是否与当前结构一致
- 输出放行建议、遗留问题和回归检查结论

## 5. 不可破坏约束

### 5.1 运行约束

重构过程中必须保持以下能力不受影响：

- `stock_trade_demo/web_app.py` 仍可作为 Web 入口
- `stock_trade_demo/choose_stock.py` 仍可作为单策略 CLI 入口
- `stock_trade_demo/compare_strategies.py` 仍可作为多策略 CLI 入口
- 当前缓存、主数据、离线策略产物的读取路径不被破坏

### 5.2 数据约束

- 不得覆盖、删除、重建、回退 `data/live_trades.csv`
- 不得在本期迁移主数据 `stock_trade_demo/stock_data.csv`
- 不得因目录美化而引入“脚本路径失效”或“缓存找不到”的问题

### 5.3 执行约束

- 优先创建新目录与新规范，不先做大规模物理迁移
- 所有移动动作都要有迁移清单与回滚依据
- 先整理“认知层”，再整理“物理文件层”，最后才考虑“代码引用层”

## 6. 目标目录治理模型

### 6.1 顶层建议治理模型

```text
quant/
├── docs/           # 正式文档、治理方案、架构说明
├── data/           # 共享基础数据、宏观数据、受保护实盘数据
├── scripts/        # 顶层构建、下载、审计脚本
├── strategy/       # 现有择时离线产物目录（短期保留）
├── output/         # 通用运行输出
├── reports/        # 汇总报告、人工分析、过程性记录
├── archive/        # 归档的历史材料与不再活跃文件
├── experiments/    # 跨项目实验
```

## 7. 文件分类规则

### A. 源代码

- `stock_trade_demo/strategies/`
- `stock_trade_demo/services/`
- `stock_trade_demo/engine/`
- `stock_trade_demo/timing/`
- `stock_trade_demo/web/`
- `scripts/`

### B. 共享基础数据

- `data/`
- 特例保留：`stock_trade_demo/stock_data.csv`、`stock_trade_demo/stock_data.parquet`

### C. 缓存

- 一律进入 `.cache/`
- 可重建
- 不作为正式文档或研究报告引用源

### D. 离线策略产物

- 现阶段继续保留在 `strategy/`
- 通过 README 明确其语义为“产物目录，不是源码目录”

### E. 报表与图表

- 统一进入 `reports/`
- 子项目专属报表优先进入 `stock_trade_demo/reports/`

### F. 实验脚本

- 统一进入 `experiments/`
- 已证伪/历史留档脚本可进入 `archive/`

### G. 临时文件

- 统一进入 `tmp/` 或 `archive/tmp_scripts/`
- 不长期停留在主目录

### H. 正式文档

- 顶层通用文档进入 `docs/`
- 子项目专属文档进入 `stock_trade_demo/docs/`

## 9. 审查机制

PM 对各角色成果按以下维度审查：

1. **完整性**：是否覆盖了要求范围
2. **风险控制**：是否遵守不可破坏约束
3. **可维护性**：整理后是否更清晰、更统一
4. **可验证性**：是否能解释迁移前后关系
5. **文档化程度**：是否充分记录过程与决策
